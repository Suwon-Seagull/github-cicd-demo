# Git Convention

이 문서는 Git을 사용하여 협업할 때 일관성을 유지하기 위한 규칙들을 정의합니다.

---

## 📝 커밋 메시지 구조

커밋 메시지는 다음 구조를 따릅니다.

```
<type>(<scope>): <subject>
<BLANK LINE>
<body>
<BLANK LINE>
<footer>
```

### 1. Type

커밋의 종류를 나타냅니다. 다음 중 하나를 사용합니다.

*   **feat**: 새로운 기능 추가
*   **fix**: 버그 수정
*   **docs**: 문서 수정
*   **style**: 코드 포맷팅, 세미콜론 누락, 코드 변경이 없는 경우
*   **refactor**: 코드 리팩토링
*   **test**: 테스트 코드 추가, 리팩토링
*   **chore**: 빌드 업무 수정, 패키지 매니저 설정

### 2. Scope (선택 사항)

커밋이 영향을 미치는 범위를 나타냅니다. (예: `feat(api): ...`)

Scope는 정해진 규칙이 없으며, 프로젝트의 구조에 따라 자유롭게 설정할 수 있습니다. 일반적으로 아래와 같은 항목들이 사용됩니다.

*   **모듈 또는 컴포넌트 이름**: `api`, `db`, `ui`, `server`
    *   `fix(api): ...`
*   **기능 또는 페이지 이름**: `auth`, `search`, `profile`
    *   `feat(auth): ...`
*   **설정 또는 도구 이름**: `build`, `ci`, `webpack`, `gradle`
    *   `chore(ci): ...`
*   **문서 이름**: `readme`, `contributing`
    *   `docs(readme): ...`

### 3. Subject

커밋의 제목입니다. 50자 이내로 간결하게 작성하며, 마침표를 찍지 않습니다.

### 4. Body (선택 사항)

커밋의 상세 내용을 작성합니다. 어떻게 변경했는지보다 무엇을, 왜 변경했는지를 설명합니다.

### 5. Footer (선택 사항)

이슈 트래커 ID 등을 포함할 때 사용합니다. (예: `Closes #123`)

## 🌿 브랜치 전략

**Git Flow** 전략을 기반으로 하며, 다음과 같은 브랜치들을 사용합니다.

*   **main**: 제품으로 출시될 수 있는 브랜치
*   **develop**: 다음 출시 버전을 개발하는 브랜치
*   **feature**: 기능을 개발하는 브랜치 (feature/기능이름)
*   **release**: 이번 출시 버전을 준비하는 브랜치
*   **hotfix**: 출시 버전에서 발생한 버그를 수정하는 브랜치

### 브랜치 생성 규칙

*   `feature` 브랜치는 `develop` 브랜치에서 생성합니다.
*   `release` 브랜치는 `develop` 브랜치에서 생성합니다.
*   `hotfix` 브랜치는 `main` 브랜치에서 생성합니다.

## 💬 커밋 메시지 예시

### ✨ feat: 새로운 기능 추가

**헤더만 있는 경우**
```
feat: 소셜 로그인 기능 추가
```

**Scope을 포함하는 경우**
```
feat(api): 사용자 정보 조회 API 추가
```

**Body와 Footer를 포함하는 경우**
```
feat(server): 인증을 위한 JWT 토큰 발급 기능 추가

- 사용자가 이메일과 비밀번호로 로그인을 요청하면, 서버는 유효성을 검사한 후 JWT 토큰을 발급합니다.
- 토큰에는 사용자 ID와 권한 정보가 포함되며, 유효 기간은 1시간입니다.

Closes: #45
```

### 🐛 fix: 버그 수정

**헤더만 있는 경우**
```
fix: 로그인 시 비밀번호 오류 수정
```

**Scope을 포함하는 경우**
```
fix(db): 사용자 데이터 중복 저장 문제 해결
```

**Body와 Footer를 포함하는 경우**
```
fix: 메인 페이지 상품 목록 조회 오류 수정

- 특정 카테고리 선택 시, 관련된 상품 목록이 정상적으로 나타나지 않던 문제를 해결했습니다.
- API 요청 시 전달되는 쿼리 파라미터가 누락되어 발생한 문제였습니다.

Resolves: #72
Related to: #68
```

### 📚 docs: 문서 수정

```
docs: README.md 파일에 프로젝트 실행 방법 추가
```

### 🎨 style: 코드 스타일 변경

```
style: 코드 전체에 Prettier 포맷팅 적용
```

### ♻️ refactor: 코드 리팩토링

```
refactor(service): 중복된 인증 로직을 AuthUtils로 분리
```

### ✅ test: 테스트 코드 관련 변경

```
test(service): 사용자 서비스에 대한 단위 테스트 코드 추가
```

### 🚚 chore: 빌드 및 기타 변경

```
chore: 의존성 관리를 위해 Gradle 버전 8.0으로 업데이트
```
